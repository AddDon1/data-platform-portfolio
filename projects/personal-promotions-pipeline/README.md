## 🛍️ Персональные предложения месяца (ППМ)  
### ML-пайплайн для рекомендаций 2+ млн клиентов ритейлеру  
📊 Вовлечённость: с 0.81% до 7.51% (+827%), AUC-ROC = 0.83, охват = 99.2%  
🛠️ CatBoost, ClickHouse, MariaDB, API интеграции, чанкинг  

---

### 🧩 **Проблема**  
Крупный ритейлер должен **ежемесячно генерировать персональные акции** для 2+ млн клиентов.  
Без ML — только ручные правила или упрощённые фильтры (например, «тот, кто покупал редко категорию X»),  
→ но это приводит к **высокому долю неактуальных рекомендаций**,  
→ и **низкому охвату** — только 60–70% клиентов получали предложения.

**Бизнес-требования:**  
- Сделать рекомендации **персональными и актуальными**  
- Достичь **охвата ≥95%** (даже для новых/редких клиентов)  
- Увеличить **CTR** на ≥15%  

---

### 🛠️ **Решение**  
Я построил **end-to-end ML-пайплайн**, который:
1. **Собирает признаки** за 12 месяцев: временные, поведенческие, товарные, демографические  
2. **Формирует таргет**: целевая категория = (а) редко покупаемая ИЛИ (b) высокая сезонность  
3. **Обучает CatBoost**: `scale_pos_weight` для баланса, GPU-ускорение, ранняя остановка  
4. **Делает инференс** в чанках (по 100k клиентов), загружает в БД  
5. **Генерирует fallback** для «холодных» клиентов (по правилам: ABC + seasonality)  
6. **Интегрируется** в внешнюю систему через API → 9 категорий с атрибутами: название, картинка, кэшбек, описание  

#### Реализация:  
- **CatBoostClassifier** (GPU, 20+ cat-признаков, One-Hot encoding для числовых категорий)  
- **ClickHouse** → ежемесячное обновление признаков (за 3 мес + 1 мес в прошлом году) 
- **MariaDB** → таблицы для `predict`, `fallback_rules`  
- **Airflow** → ежемесячный запуск пайплайна  
- **Внешний API** → импорт JSON-файлов (по 6000 записей/файл)  

**Масштаб:**  
- 2.1 млн клиентов → 14 чанков  
- Инференс на 1 GPU (RTX 3090) → 45 мин  
- Всё: от данных до API — **за 4 часа**  

---

### ✅ **Результаты**  

| Метрика | До ML | После ML | Улучшение |
|---------|-------|----------|-----------|
| **AUC-ROC** | — | **0.83** | — |
| **Охват рекомендаций** | ~65% | **99.2%** | **+34.2 п.п.** |
| **Вовлечённость** | 0.81% | **7.51%** | **+827%** |
| **Precision** | 58% (ручные правила) | **83.1%** | **+25 п.п.** |
| **Время подготовки** | 3 дня | **45 мин** | **↓90%** |

**Бизнес-эффекты:**  
- ✅ Клиенты получают **персональные предложения**, а не "массовые"  
- ✅ Увеличение **конверсии в акциях** и **среднего чека**  
- ✅ **Снижение оттока** среди клиентов с «холодными» аккаунтами  
- ✅ **Экономия времени**: не 3 дня ручной работы, а 45 мин автопайплайна  
- ✅ Увеличение **разнообразия в корзине клиента** и **мотивации покупать то, что приобретается у конкурентов**

### 🧰 **Технологии**  
- **Python**: pandas, scikit-learn, CatBoost  
- **ClickHouse**: хранение чеков, товарных и поведенческих признаков  
- **MariaDB**: промежуточные таблицы для инференса и финального списка  
- **Внешний API**: импорт JSON-сегментов  

---

> 💡 **Почему не XGBoost/LightGBM?**  
> Я выбрал **CatBoost** потому что:  
> - Встроенный `scale_pos_weight` для несбалансированных классов (target=1 ≈ 15%),  
> - Стабильность на One-Hot-encoded категориальных признаках (без manual label-encoding),  
> - GPU-ускорение в 3× по сравнению с CPU (обучение 2 ч → 40 мин).  

> 💡 **Почему чанкинг по 100k?**  
> Обработка всех 2.1 млн сразу — **выход за RAM** на стейджинге (8 GB).  
> Чанкинг позволяет:  
> - Использовать любой сервер,  
> - Легко масштабироваться (добавляем CPU),  
> - Не падать при одной ошибке — обрабатываем только 1 чанк.
