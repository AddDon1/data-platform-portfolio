## 🎉 Механизмы лояльности: автоматизация акций и лотерей

### 🧩 Проблема  
Руководство 100+ магазинов заказывало **еженедельные акции** (например, «Сладкий холод»):  
- Клиент должен получить **1 купон/шанс** при покупке от 800 ₽ или при наличие товаров-участников или за каждую первую покупку на 3 день покупок в неделю,  
- Но **не более 5 чеков в сутки** (чтобы стимулировать повторные посещения).  
- Аналитики тратили **4.5 часа в неделю** на:  
  • Выгрузку данных из ClickHouse  
  • Расчёт шансов вручную  
  • Импорт в БД  
  • Формирование Excel-файлов участников  

**Без автоматизации:**  
- ❌ Ошибки в расчётах (в среднем 12% нарушений логики)  
- ❌ Задержки: купоны выдавались не вовремя  
- ❌ Нет отчётов по ЦВП (119 победителей по городам)  

---

### 🛠️ Решение  
Создал **универсальный пайплайн**, который:  
1. **Фильтрует чеки** по:  
   - магазинам (`format_name = 'Фрешмаркет'`)  
   - сумме (`summ >= 800`)  
   - периоду (`START_DATE` → `END_DATE`)  
2. **Рассчитывает шансы**:  
   - `koef_sum = floor(summ / 800)` — сколько шансов за сумму  
   - `koef_qnt = floor(quantity)` — сколько шансов за количество товаров  
   - `cnt = least(koef_sum, koef_qnt)` — итоговое кол-во шансов (минимальный показатель)  
3. **Исключает дубли за день**:  
   - `ROW_NUMBER() OVER (PARTITION BY phone, toYYYYMMDD(close_date))`  
   - Но для механики «3 день покупок в неделю» — `day_rank % 3 = 0`  
4. **Интегрируется в 3 шага**:  
   - Сохранение в MariaDB  
   - Импорт в API МП
   - Формирование Excel: `участники_{date}.xlsx` с 8+ колонками для розыгрышей и аналитики 

#### Реализация:  
- **ClickHouse (основная выборка)**:  
  ```sql
  ROW_NUMBER() OVER(PARTITION BY phone, toYYYYMMDD(close_date) ORDER BY close_date) AS row_num  
  -- Ограничивает до 5 чеков в день  
  HAVING s.row_num < 6

  -- Далее, для акции "3 покупки в неделю":
  SELECT DISTINCT 
      clc.person_uid,
      toStartOfWeek(chl.close_date, 9) AS week_start,
      toDate(chl.close_date) AS purchase_date
  FROM ...
  WHERE ...
  
  -- Внутри: 
  ROW_NUMBER() OVER (PARTITION BY person_uid, week_start ORDER BY purchase_date) AS day_rank
  WHERE day_rank % 3 = 0 AND day_rank > 0

  -- Расчёт шансов:
  SELECT 
      ...,
      floor(sum((s.summ) / 800)) AS koef_sum, 
      floor(sum((s.quantity) AS koef_qnt,
      least(floor(sum((s.summ) / 800)), 1) AS cnt
  -- => макс. 1 купон на чек (даже при summ ≥ 1600)
  ```
- **Python (pandas):**
`groupby(['phone', 'день']).idxmin()` → не используется в этой реализации
Вместо этого: `ROW_NUMBER() в ClickHouse + HAVING row_num < 6`
`floor(summ / 800)` → расчёт koef_sum, но не для количества товаров (товаров не учитывается)
Дублирование строк: `for _ in range(cnt - 1): ...` — создаёт по строке на купон
Для второй выборки: `counter = XXXXXXXX` — фиксированный ID для «3 покупки в неделю»

- **Excel: openpyxl** → авто-форматирование, переносы текста, высота заголовка 45 пт
- Запускается каждые 30 минут

**🧰 Технологический стек**
- ClickHouse: window-функции
- Python (pandas): `groupby().idxmin(), openpyxl`
- MariaDB: `хранимые процедуры для main`
- API: `requests + JWT (Loymax, arm)`

# 🔧 Лучшие практики: Автоматизация акций лояльности

## 📌 Общие принципы
- **Добавил** логирование всех критических этапов (используя `logger.info`, `logger.error`)
- **Использовал** конфигурацию (`config.py`, `action_params.py`) вместо хардкода дат, лимитов, ID акций
- **Разделил** логику:  
  - `query_...()` → только получение данных  
  - `process_...()` → только преобразование (pandas)  
  - `insert_...()` / `upload_...()` → только запись в БД/FTP/API  
- **Применил** `try-except` + `exit(1)` для всех критических операций — чтобы избежать частичной выгрузки и «полуошибочных» данных

## 🔧 Работа с ClickHouse
- **Добавил** `settings={'max_query_size': 99999999999}` для больших запросов — избежал `QueryTooLarge`
- **Применил** `ROW_NUMBER() OVER (PARTITION BY phone, toYYYYMMDD(close_date) ORDER BY close_date)` для ограничения до 5 чеков в день (`HAVING s.row_num < 6`)
- **Реализовал** фильтрацию «3 покупки в неделю» через `day_rank % 3 = 0`
- **Использовал** `floor(summ / 800)` и `least(...)` для расчёта шансов 
- **Сохранил** промежуточные датафреймы в `reports/debug_*.csv` для отладки

## 🌐 Работа с API Loymax / arm
- **Обработал** ответы в HTML-обёртке через `BeautifulSoup + json.loads(soup.text)`
- **Использовал** `requests.Session()` для повторных запросов (сохранение токена)
- **Разбил** массовую выгрузку на чанки по 10 000 — избежал `504 Gateway Timeout`
- **Добавил** retry-логику: `while True: try/except + time.sleep(2)` — это гарантировало стабильность загрузки купонов

## 🗄️ Работа с MariaDB
- **Настроил** подключение с `charset='utf8mb4'` и `connect_timeout=99999` — избежал сбоев при unicode
- **Применил** хранимые функции (`ins_katim_v_leto_3`) — централизовал бизнес-логику и защиту от дублей
- **Проверил** уникальность через `f"{id}_{card}_{date}"` — гарантировал, мультиначисление шансов в ондном чеке, присвоим уникальные порядковые номера в id

## 📊 Работа с Excel (openpyxl)
- **Использовал** `pd.ExcelWriter(..., engine='openpyxl')` для корректной работы с листами
- **Применил** форматирование: `wrap_text=True`, высота заголовка 45 пт — улучшил читаемость
- **Сохранил** файлы сначала в `reports/`, потом скопировал на сетевой диск — избежал ошибки *«файл занят»*

## 📂 Работа с файлами и FTP
- **Добавил** `lstrip("/")` при загрузке из FTP — избежал ошибок с путями
- **Сохранил** `ncard3.csv` (карты + дата регистрации) — использовал для фильтрации домашнего/тестового использования
- **Создал** отдельные файлы для аудита: `coupons_data.txt`, `dict_new_ready_coupons_cards.txt`, `ready_coupons.txt`

## 📝 Логирование и отладка
- **Добавил** логирование: начало/конец этапа, количественные метрики (`f"Новых купонов: {len(new_df)}"`)
